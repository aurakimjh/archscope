## 종합 판정: `CONDITIONAL`

설계의 보안 골격은 **견고하다.** 위협 모델(§11.0)이 행위자·능력·1차 방어를
명시하고, "기본 안전"을 재사용 가능한 자격증명 0건이라는 검증 가능한 정의로
고정했으며, 적대적 테스트 행렬(§11.7, SEC-1~15)이 각 방어선에 확인 수단을 붙였다.
CA 개인키 취급(무동봉·무내보내기·머신 고유·1년 만료), 상주 승격 프로세스 금지,

제품에서 흔히 빠지는 방어선을 정확히 짚고 있다.

**P0(치명) 결함은 없다.** 다만 아래 P1 1건과 P2 6건은 게이트 `PASS` 전에
닫혀야 한다. 특히 (1) 은닉 캡처 금지 원칙의 CLI 채널 미봉과 (2) 이미 구현된
리댁션 코드의 구조화-필드 대 자유텍스트 규칙 불일치는 **명시된 보안 계약(§11.1,
SEC-1/2)과 직접 충돌**하므로 우선 해소해야 한다.

## Findings

### P1-1 — 은닉 캡처 금지(§11.1)가 CLI/헤드리스 채널에서 강제되지 않는다

- **근거:** §11.1은 "은닉 불가 — 캡처 중 항상 화면 표시, 백그라운드 무표시 캡처
  모드 미제공, 트레이 아이콘 상태 변경"을 핵심 방어선으로 둔다. 그러나 이 방어선의
  실효 수단이 **GUI 표시에만** 결부돼 있다. §11.1 마지막 항은 "CLI에 캡처 시작
  명령을 추가한다면 동일 고지와 명시 플래그를 요구"라고만 적고, §14 열린 질문 7은
  **"CLI 캡처 명령을 만들 것인가"를 미결로 남겼다.** 명시 플래그·1회 고지만으로는
  헤드리스로 시작된 캡처에 **지속적 실행 표시(항상 보이는 인디케이터)가 없다** —
  "무표시 캡처 모드 미제공" 불변식을 CLI가 그대로 위반할 수 있다.
- **영향:** 시스템 전역 HTTP 캡처는 §11.1 자체 표현대로 "도청 도구"다. 표시 없는
  백그라운드 캡처 경로가 하나라도 열리면, 이 제품의 최상위 오남용 방어선이
  무력화된다(타인 동의 없는 캡처). 스크립트/스케줄러로 상시 캡처를 걸어두는 것을
  막을 계약이 없다.
- **권고:** Phase 2 착수 **전에** 다음 중 하나를 계약으로 고정한다.
  1. CLI에서 캡처 **시작을 제공하지 않는다**(가져오기·분석만). — 가장 단순하고
     방어선과 일관.
  2. 제공한다면 (a) 완전 분리(detached/데몬화) 실행 거부, (b) 실행 내내 stderr에
     지속 배너 출력, (c) 로컬 감사 레코드(시작 시각·대상·사용자) 강제, (d) 1회가
     아닌 세션마다 고지 확인. 이를 §11.7 행렬에 `SEC-16`(비대화형 은닉 캡처 시도
     → 거부/표시)로 추가한다.
- **비고:** 이 항목은 §14 Q7이 미결이라는 사실 자체가 게이트 리스크다. 개별 보안
  게이트가 존재하는 이유가 바로 이런 미봉을 강제 종료하기 위함이다.

### P2-1 — CA 개인키·평문 바디의 메모리 위생 주장이 Go 런타임에서 구조적으로 보장 불가

- **근거:** §11.0 위협 모델은 "크래시 덤프·로그 수집" 행위자에 대해 "키 즉시 폐기,
  덤프에서 캡처 버퍼 제외"를, §11.2는 "사용 후 즉시 폐기. 로그·크래시 덤프에 절대
  포함 금지"를, SEC-10은 "캡처 중 강제 크래시 후 덤프에 CA 개인키·평문 바디
  미포함"을 통과 기준으로 둔다. 그러나 엔진은 Go로 작성된다(교차 검증 코드 전부
  Go). **GC 관리 메모리에서는 `[]byte`/`crypto/ecdsa` 키 material의 신뢰할 수 있는
  제로화가 불가능**하고(가비지 컬렉터가 복사·이동한 잔본을 남길 수 있음), 프로세스
  코어 덤프에서 힙 영역을 배제하는 것도 기본적으로 되지 않는다. "즉시 폐기"는
  Go에서 참조 해제 이상을 의미하기 어렵다.
- **영향:** SEC-10은 현재 서술대로라면 **통과를 보장할 수 없는 수용 기준**이다.
  명세와 런타임 현실의 간극이 게이트 통과 판정을 흐린다.
- **권고:** 보장 문구를 구체 메커니즘으로 치환한다. 예: (a) 프로세스 코어 덤프
  비활성(Linux `RLIMIT_CORE=0`/`PR_SET_DUMPABLE=0`, Windows WER 제외 등록),
  (b) CA 개인키를 앱 프로세스 힙에 평문으로 상주시키지 않고 **OS 키스토어 내에서만
  서명 수행**(가능한 플랫폼), (c) 불가피한 평문 보관 구간은 `mlock`/`VirtualLock`
  버퍼로 한정하고 스왑 배제. SEC-10 통과 기준을 이 메커니즘의 존재 여부로 재정의.

### P2-2 — §6.5.6 리댁션 기법 표가 §11.3.1의 철회 결정과 모순

- **근거:** §11.3.1(2026-07-19 개정)은 세 기법을 안전하지 않다고 판단하고 기본값을
  전면 삭제로 바꿨다 — JWT는 **payload 전체 삭제**, 쿠키는 **완전 삭제**, blob
  식별자는 **랜덤 opaque ID**. 그런데 §6.5.6의 "분석 가치를 보존하는 리댁션 기법"
  표(문서 라인 1508–1513)는 여전히 **"JWT 서명만 제거 — 클레임은 계속 읽힌다"**와
  **"쿠키 값을 해시 접두로 치환"**을 권장 기법으로 나열한다. 두 절이 서로 반대되는
  기본값을 지시한다.
- **영향:** §6.5.6만 읽고 구현하면 JWT payload(이메일·subject·tenant·role·내부
  식별자)와 무염 해시 쿠키가 그대로 남아 **SEC-1을 실패한다.** 실제 구현
  (`redact.go`)은 §11.3.1을 올바르게 따르지만(JWT 전체 `[REDACTED_JWT]`, 쿠키
  `[REDACTED]`), 문서가 상반된 지침을 병존시키면 후속 구현·회귀에서 약한 쪽으로
  되돌아갈 위험이 상존한다.
- **권고:** §6.5.6 표의 JWT·쿠키 행에 "**기본값은 §11.3.1로 대체됨. 아래는
  옵트인 상관관계 보존 시에만**"을 명시하거나 해당 행을 §11.3.1 참조로 축약한다.

### P2-3 — CA at-rest 폴백 권한이 POSIX 모드로만 서술됨 (크로스 플랫폼 미완)

- **근거:** §11.2는 파일 폴백 시 "0600"을, §11.0의 "공용 워크스테이션 타 사용자"
  방어선(SEC-11)은 "OS 권한으로 차단"을 든다. 그러나 `0600`은 POSIX 모드값이다.
  Windows(NTFS ACL)와 macOS의 상위 디렉터리 모드(예: `~/Library` 0700 전제)에
  대응하는 **폴백 ACL 계약이 명세되지 않았다.** DPAPI/키체인/Secret Service
  실패 시의 파일 폴백이 각 플랫폼에서 실제로 타 로컬 사용자에게 불가독인지가
  "0600"만으로는 판정되지 않는다.
- **영향:** SEC-8/SEC-11의 통과 기준이 플랫폼별로 검증 불가능. Windows 폴백 경로가
  기본 상속 ACL로 생성되면 타 사용자 가독 위험.
- **권고:** 폴백 저장의 크로스 플랫폼 ACL 계약을 명시(Windows: 소유자 전용 DACL,
  상속 차단; macOS/Linux: 0600 + 상위 디렉터리 0700). §11.7에 폴백 경로 권한
  실측 항을 추가.

### P2-4 — Linux/다중 프로필 CA 신뢰 저장소 열거 완전성이 미명세 (SEC-12의 실질 난점)

- **근거:** §11.2.1은 "제거는 각 저장소별 개별 보고, 하나라도 남으면 상태는
  `trusted`"라는 훌륭한 규칙을 두고 SEC-12는 "모든 저장소에서 제거"를 요구한다.
  §9.1은 시스템 저장소와 NSS DB(Chrome/Firefox)가 따로 논다는 점까지 지적한다.
  그러나 **"모든 저장소"의 열거 범위**가 정의되지 않았다 — Linux에서 NSS DB는
  사용자별·브라우저별·**프로필별**로 존재하며(예: `~/.mozilla/firefox/<profile>`
  다수, `~/.pki/nssdb`), 기본 프로필만 스캔하면 비기본 프로필에 등록된 CA가
  "제거 완료" 판정 뒤에도 남는다.
- **영향:** "제거했다고 생각했지만 남아 있는 CA" — §11.2.1이 스스로 최악
  시나리오로 지목한 상태가 다중 프로필에서 실제로 발생. SEC-12를 형식적으로만
  통과.
- **권고:** 제거·검증이 열거해야 할 저장소 집합을 플랫폼별로 명시(시스템 앵커 +
  발견된 모든 NSS DB 경로 열거). 열거 불가/실패 저장소는 §11.2.1 규칙대로 수동
  절차와 함께 명시 보고하고 상태를 `none`으로 내리지 않는다.

### P2-5 — (구현) 리댁션 규칙이 구조화 필드와 자유텍스트에서 서로 다른 denylist를 쓴다

- **근거 (코드 대조):** `redact.go`에는 두 종류의 민감 키 판정이 있다.
  - 구조화 경로(쿼리 키, JSON 키, 폼 키, 쿠키): `sensitiveKey → sensitiveNormalizedKey`
    (redact.go:349–366) — **넓은 집합**: `token, accesstoken, refreshtoken, apikey,
    session, code, secret, signature, password, ssn, creditcard …` + `*token/*secret/
    *signature/*credential` 접미사.
  - 자유텍스트 경로(평문 바디, URL 프래그먼트, **비-denylist 헤더 값**):
    `applyText → assignmentPattern`(redact.go:65) — **좁은 집합**:
    `password|access_token|refresh_token|api_key|service_key|client_secret|
    private_key|credential|secret|signature|credit_card|ssn`. 여기엔 **`token`(단독),
    `session`, `code`, `auth`, 일반 `*token` 접미사가 빠져 있다.**
  - 자유텍스트 경로가 실제로 도달하는 지점: `RedactBody`의 비-JSON·비-폼 텍스트
    바디(redact.go:209), `RedactURL`이 프래그먼트를 포함한 전체 URL에 거는
    `applyText`(redact.go:143 — 쿼리 키만 넓은 `sensitiveKey`로 처리되고 **프래그먼트
    파라미터는 좁은 규칙만 통과**), 비민감 헤더 값(redact.go:155).
- **결과:** `text/plain` 바디나 URL 프래그먼트의 `token=<opaque>`,
  `sessionId=<opaque>`, `authToken=<opaque>`, `code=<opaque>`(비-JWT)는
  **리댁션되지 않는다.** 같은 키가 JSON/폼/쿼리에 있으면 넓은 집합으로 잡힌다.
  (JWT 형태 값은 키와 무관하게 `jwtPattern`이 잡으므로 `id_token=eyJ…`은 안전.
  누출되는 건 **비-JWT opaque 토큰**이다.)
- **영향:** §11.3의 "기본 켜짐·저장 시점 리댁션" 계약이 특정 채널(평문 바디·
  프래그먼트·비표준 헤더)에서 부분적으로 뚫린다. SEC-1의 "재사용 가능한 자격증명
  0건"을 이 채널들에서 보장하지 못한다. (SEC-2의 문자 그대로의 사례 —
  쿼리스트링 토큰·POST 폼 `password` — 는 통과한다.)
- **권고:** `assignmentPattern`의 키워드 교대(alternation)를 `sensitiveNormalizedKey`
  집합과 정렬한다 — 최소한 `token`, `session`, `code`, `auth`, 일반 `[\w-]*token`
  접미사를 추가. 회귀 테스트에 평문 바디·URL 프래그먼트·비표준 헤더 값 3채널의
  bare-token 케이스를 추가(현재 `redact_test.go` 미커버 여부 확인 권장).

### P2-6 — 캡처 시점 범위 제한(프로세스/도메인 스코프) 부재 (데이터 최소화)

- **근거:** 검토 항목 6에 해당. 프록시 모드는 시스템 프록시를 따르는 **모든**
  트래픽을 본다. 설계는 이를 사후에 리댁션하고 §10.1.2에서 증거에 scope를 붙이지만,
  **캡처 시점에 "선택한 프로세스/호스트만 기록"하는 allowlist/denylist가 없다.**
  §10.2 "프로세스별 설정 안내"는 사용자에게 프록시 설정을 안내하는 것이지 기록
  필터가 아니고, §5.3 allowlist는 피닝 패스스루용이다.
- **영향:** 대상이 아닌 앱·타 사용자 세션의 트래픽까지 세션 파일에 수집된 뒤
  리댁션에만 의존한다. P2-5처럼 리댁션에 빈틈이 있으면 수집 범위가 넓을수록
  부수 유출 표면이 커진다. Privacy-by-design(수집 최소화) 관점에서 방어가
  리댁션 한 겹에 집중돼 있다.
- **권고:** 캡처 시점 스코프 필터(프로세스 identity / 호스트 패턴 allowlist)를
  옵트인으로 제공하고, 범위 밖 트랜잭션은 **저장 전에 드롭**한다(메타데이터도
  남기지 않거나, 남기더라도 바디·헤더 미수집). 이는 §11.0 "기본 안전"과 §11.1
  고지·동의 모델을 강화한다. §11.7에 스코프 밖 트래픽 미저장 검증 항 추가.

### P3-1 — 리프 인증서 개인키의 비영속성이 명시되지 않음

- **근거:** §5.1은 호스트별 리프 인증서를 메모리 캐시(유효기간 2일)한다고만 하고,
  §11.2는 **CA** 키 보관만 다룬다. 리프 개인키가 디스크에 절대 기록되지 않는지가
  명시적 계약이 아니다.
- **권고:** "리프 개인키는 메모리 상주·세션 종료 시 폐기, 디스크 미기록"을 §5.1/
  §11.2에 명문화. (CA와 동일한 폐기 한계는 P2-1에 종속.)

### P3-2 — WebSocket/SSE 메시지: 설계는 "리댁션 후 보존", 구현은 "전면 드롭" (안전하나 회귀 위험)

- **근거:** §6.5.6은 `_webSocketMessages`가 Chrome 소독에서 제외되므로 **가져오기
  경로에서 리댁션**하라고 강조한다(§11.3의 전제를 바꾸는 항목으로 명시). 그러나
  현재 파서 `harEntry`(parser.go:79–)는 `_webSocketMessages`/`_eventSourceMessages`를
  **역직렬화하지 않는다** — 즉 WS/SSE 페이로드는 출력에 도달하지 않고 통째로
  버려진다. **현 상태는 안전**(누출 0)하지만 설계 의도(보존+리댁션)와 다르다.
- **영향:** 지금은 문제없음. 다만 향후 WS 분석 가치를 위해 파싱을 추가할 때
  리댁션 배선을 빠뜨리면 §6.5.6이 경고한 정확한 취약점이 조용히 재도입된다.
- **권고:** "WS/SSE 콘텐츠는 리댁션 없이 출력에 도달하지 않는다"를 불변식으로
  두고, WS/SSE 필드에 원문 비밀을 넣은 HAR을 가져와 출력에 부재함을 확인하는
  회귀 테스트를 추가(SEC-6의 하위 케이스).

### P3-3 — 세션 보존 기본값과 삭제 의미

- **근거:** §11.5는 기본 보존 7일·자동 삭제 옵트아웃("삭제를 안내"), "세션 삭제는
  blob 포함 완전 삭제"를 둔다. "완전 삭제"는 unlink이며 secure erase가 아니다.
  리댁션이 기본 켜짐이라 디스크 잔본의 민감도는 낮으나, 리댁션을 끈 세션(§11.3
  옵트아웃)은 평문 바디가 unlink 후에도 매체에 잔존할 수 있다.
- **권고:** (1) 리댁션을 끈 세션은 보존 안내를 더 강하게(또는 리댁션 off 시
  기본 보존 단축), (2) "완전 삭제 = 참조·blob unlink이며 매체 수준 소거는
  아니다"를 문서에 정직하게 한정(§11.3.2의 "손상 탐지 ≠ 무결성"과 같은 태도).

## §11.7 적대적 테스트 행렬(SEC-1~15) 독립 점검

| ID | 설계 통과 기준 | 독립 판정 | 비고 |
|---|---|---|---|
| SEC-1 | 재사용 자격증명 0건, JWT payload 미포함 | **부분** | 구조화 필드 OK. 평문/프래그먼트 bare-token 누출(P2-5) |
| SEC-2 | 쿼리 토큰·POST 비밀번호 리댁션 | 통과(문자 그대로) | `password`/쿼리 키 커버. 인접 키워드는 P2-5 |
| SEC-3 | export 후 동일 + `commandLine`/`user` 미포함 | 통과(코드상) | `RedactProcess`가 user 삭제·commandLine 리댁션·execPath basename화(redact.go:213) |
| SEC-4 | 압축폭탄·10만 entry·깊이 1000·100MiB 문자열 파싱 전 거부 | **통과** | `guard.go` 전량 커버: MaxEntries 100k, 깊이 256, gzip 비율 1000, MaxBytes 256MiB, 문자열 16MiB |
| SEC-5 | 구조 손상 fatal 중단 | 통과 | `ErrStructuralHAR`로 `log.entries` 비배열 거부(parser.go:206) |
| SEC-6 | 가져오기 경로 리댁션 적용 | 통과 | 리댁션이 파서에 무조건 배선, 비활성 경로 없음(parser.go:225). WS는 P3-2 |
| SEC-7 | catastrophic backtracking 불가 | 통과 | Go `regexp`=RE2. 규칙당 시간 상한 후 비활성화+finding(redact.go:306) |
| SEC-8~15 | CA·덤프·IPC·피닝·업스트림 | 설계만 | 미구현. SEC-10은 P2-1로 재정의 필요, SEC-12는 P2-4 |

행렬 자체의 구성은 우수하다 — 위협 모델의 각 행위자에 확인 수단이 1:1로 붙어
있고, SEC-4~7이 캡처 없이 Phase 1부터 유효하다고 정확히 못박았다. 위 판정은
"행렬이 부실"이 아니라 **"행렬이 요구하는 기준을 구현/명세가 아직 다 못 맞춤"**을
뜻한다.

## 게이트 `PASS` 조건 (권고)

1. **P1-1** 종료: CLI/헤드리스 캡처의 은닉 방지 계약을 확정(미제공 또는 강제
   표시·감사·비분리). §14 Q7을 닫고 `SEC-16` 추가.
2. **P2-5** 종료: 자유텍스트 리댁션 denylist를 구조화 필드 집합과 정렬 + 3채널
   회귀 테스트. (이미 구현된 코드의 결함이므로 우선순위 높음.)
3. **P2-1/2/3/4/6** 반영: SEC-10 메커니즘화, §6.5.6↔§11.3.1 모순 제거, 크로스
   플랫폼 CA ACL·NSS 열거 명세, 캡처 시점 스코프 필터 계약.
4. P3 3건은 문서 정직성·회귀 안전 차원에서 반영하되 게이트 차단 사유는 아님.

P0 부재 및 방어선 골격의 완성도를 고려할 때, 위 P1·P2가 닫히면 H-SEC1은 `PASS`로
전환 가능하다. 현 상태의 독립 판정은 `CONDITIONAL`이다.
